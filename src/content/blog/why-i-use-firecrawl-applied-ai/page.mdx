## The cliff nobody budgets for

Every AI project that touches web data has the same hidden cliff. The interesting work — the retrieval, the agent logic, the model prompting — assumes you already have clean text in hand. Getting clean text from a real, modern, JavaScript-rendered, occasionally bot-defended web page is the part nobody budgets for, and it is the part that pages you at 2 a.m.

My first few scraping setups were the obvious stack: a headless browser, some CSS selectors, a proxy when things got blocked. Each one worked in the demo and then slowly fell apart. A site would redesign and my selectors would snap. A page would start lazy-loading content below the fold and my fetch would return an empty shell. Cloudflare's bot protection would notice my traffic and the whole job would start failing *silently* — not erroring, just returning challenge pages that my parser dutifully turned into garbage.

The silent failure was the one that bothered me most. A loud crash you catch in CI. A scraper that returns plausible-looking-but-wrong text feeds bad data into a model, and you find out three weeks later when someone asks why the summaries are off.

What Firecrawl actually replaced, when it went into my projects, was a whole junk drawer of moving parts: a headless Chrome or Playwright instance for rendering JavaScript-heavy pages, a rotating proxy service to avoid getting blocked (plus the billing relationship that came with it), a pile of per-site selectors and cleanup logic that broke on every redesign, and the retry-backoff-rate-limit glue I wrote once, got wrong, and rewrote. All of it collapsed into one call.

```javascript
import Firecrawl from '@mendable/firecrawl-js'

const app = new Firecrawl({ apiKey: process.env.FIRECRAWL_API_KEY })

const page = await app.scrapeUrl('https://example.com/blog/post', {
  formats: ['markdown'],
  onlyMainContent: true,
})

console.log(page.markdown) // clean text, nav and footer gone
```

That `onlyMainContent: true` flag does the thing I used to write a hundred lines of selector logic for — it strips the nav, the cookie banner, the footer, the sidebar of related links, and hands back the article. When the source site redesigns, the flag keeps working, because it is not pinned to the site's markup. The whole category of problem moved behind one API call.

## What held up once it was live

Plenty of scraping tools demo well. What decided it for me was the way three properties behaved after the feature was live and getting real traffic.

The first is that it handles JavaScript without me running a browser. Most pages worth scraping today render their content with JavaScript, so a raw HTTP fetch sees an empty `<div id="root">` and nothing else. The old answer was to run a headless browser, which means you now own a browser fleet — memory leaks, zombie processes, version upgrades that break things, and a job queue to keep it all moving. Firecrawl renders the page on its side and returns the result. The infrastructure still exists; it is just not mine to babysit. I got into that tradeoff at more length in <Link href="/blog/best-web-scraping-api-2026">my comparison of scraping approaches for AI developers</Link>.

The second is the one I underrated until I used it: schema extraction means my code stops breaking on redesigns. Instead of writing `div.price > span.amount` and praying the site never touches its markup, I pass a JSON schema and a short prompt, and a model fills it in from the page.

```javascript
const result = await app.scrapeUrl('https://example.com/pricing', {
  formats: ['json'],
  jsonOptions: {
    prompt: 'Extract each plan name and its monthly price in USD.',
    schema: {
      type: 'object',
      properties: {
        plans: {
          type: 'array',
          items: {
            type: 'object',
            properties: {
              name: { type: 'string' },
              monthlyPriceUsd: { type: 'number' },
            },
          },
        },
      },
    },
  },
})
```

When the site moves the price from a `<span>` into a `<div>` next quarter, this keeps working, because it is extracting *meaning* rather than matching a CSS path. That's the difference between a scraper I check on weekly and one I forget exists. I dug into the structured-extraction workflow in <Link href="/blog/extract-structured-data-websites">extracting structured data from websites</Link>.

The third is that the output is genuinely LLM-ready — clean Markdown, structure intact, the chrome gone. That matters more than it sounds. Feeding a model raw HTML, or text with the nav and footer still glued on, wastes tokens and dilutes whatever you are trying to retrieve or summarize. Markdown with the headings preserved chunks cleanly for a vector store and reads well in a prompt. When the input layer is clean, the rest of the <Link href="/blog/firecrawl-for-rag-pipelines">RAG pipeline</Link> gets simpler, because you aren't writing defensive cleanup code between the scrape and the embedding step.

## Where I wouldn't reach for it

The honest version of any recommendation includes the cases where it's the wrong call, and there are a few.

When I need to scrape behind a login with complex, stateful session handling — multi-step auth, CSRF tokens, flows that depend on prior clicks — I reach for a self-hosted browser automation setup where I control the session end to end. The same goes for anything that needs fine-grained control over browser config, custom headers tied to a specific auth scheme, or behavior that depends on interacting with the page across several steps. Owning the browser is the right answer there.

For a single static HTML site with a stable structure, a managed API is overkill; Beautiful Soup or Cheerio over a plain fetch is faster and free. And if scraping were my core product rather than plumbing for an AI feature, I'd think hard about owning the stack, the way I did when I ran one. That calculus is the whole subject of <Link href="/blog/firecrawl-vs-crawl4ai">Firecrawl vs Crawl4AI</Link>, where the open-source, self-hosted path makes real sense for some teams.

The line I draw is simple. When web data is the input to a feature, I want it to be a function call. When web data *is* the feature, I want to own it.
