Searching email shouldn't require a server. Here's how I built gmail-mirror, a local Go CLI that syncs your Gmail to SQLite with FTS5 for instant, sub-100ms full-text search.

## The Problem

Gmail's search is fine for simple queries, but:
- Can't search across 15 years of email efficiently
- No offline access
- Labels and filters don't help with complex research
- API rate limits kill bulk searches

I needed to search my own archive — 29,250+ messages — instantly, locally, with complex boolean queries.

## The Solution: gmail-mirror

A Go CLI that:
1. Uses Gmail API to sync all messages
2. Stores in SQLite with FTS5 for full-text search
3. Provides a fast CLI interface

### Installation

```bash
git clone https://github.com/zackproser/gmail-mirror
cd gmail-mirror
go build -o gmail-mirror
./gmail-mirror sync  # Initial sync (takes ~10 min for 30k messages)
```

### Searching

```bash
# Simple search
./gmail-mirror search "expert network"

# With filters
./gmail-mirror search --from "katia" --after "2025-01-01" --limit 50

# Full-text with boolean
./gmail-mirror search "urgent AND (meeting OR call)"

# JSON output for scripting
./gmail-mirror search "invoice" --json --limit 10
```

### Performance

- **29,250 messages** indexed
- **Sub-100ms** query time for full-text searches
- **~500MB** SQLite database (including full text)

### Database Schema

```sql
-- Messages table
CREATE TABLE messages (
    id TEXT PRIMARY KEY,
    thread_id TEXT,
    subject TEXT,
    from_addr TEXT,
    to_addr TEXT,
    date TEXT,
    body_text TEXT,
    body_html TEXT
);

-- FTS5 virtual table for full-text search
CREATE VIRTUAL TABLE messages_fts USING fts5(
    subject,
    from_addr,
    body_text,
    content='messages',
    content_rowid='rowid'
);
```

## Why FTS5?

SQLite's FTS5 provides:
- **Tokenizer support** — snowball, porter stemmer
- **Prefix searches** — "prog*" matches "programming"
- **Boolean operators** — AND, OR, NOT
- **Relevance ranking** — BM25 scoring

```sql
-- FTS5 query with ranking
SELECT messages.*, bm25(messages_fts) as rank
FROM messages_fts
JOIN messages ON messages.rowid = messages_fts.rowid
WHERE messages_fts MATCH 'urgent AND meeting'
ORDER BY rank
LIMIT 10;
```

## Incremental Sync

The magic: only sync what's new.

```bash
# First run: full sync
./gmail-mirror sync

# Subsequent runs: only new messages
./gmail-mirror sync  # Takes seconds, not minutes
```

The sync process:
1. Fetch list of message IDs from Gmail since last sync
2. Batch fetch message bodies (100 at a time)
3. Insert into SQLite, updating existing rows
4. Rebuild FTS index for new content
