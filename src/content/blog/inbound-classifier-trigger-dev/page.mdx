## Safety First: The Graduation Ladder

Here's the design constraint I started with: auto-replying to everything is a disaster waiting to happen. The failure mode — confident, context-blind form letters going out under your name — is worse than occasionally spending thirty seconds on a low-stakes reply myself.

So the system started fully manual. Every classifier output generated a draft I reviewed and clicked Send on. That manual phase wasn't a limitation — it was the test bed. Each category accumulated a corpus of confirmed-correct drafts. I was building a track record.

Once a category proved itself — roughly ten verified-correct outputs in a row, voice rules locked in — it graduated to opt-in auto-send for that specific category. Other categories stayed manual until they earned the promotion. Graduation per-category, not system-wide.

The reversion path is one env var edit and a redeploy. If anything breaks, I flip the switch and everything goes back to manual review on the next run. No code changes, no panic.

This is more interesting than "AI drafts emails." It's a human-in-the-loop architecture with an explicit graduation ladder per category.

## Google Cloud Setup: How the Scheduled Task Gets Inbox Access

Here's the part nobody writes about: how a Trigger.dev task actually gets read access to your Gmail.

### OAuth Client Type: Desktop App

The alternative (Web application) needs a hosted callback URL — something like `https://myapp.ngrok.io/callback`. That's fine if you're running a web app, but a scheduled background worker doesn't have one. Desktop app clients accept `http://localhost:<port>` as a valid callback, which works from a one-shot local script. Less infra to maintain.

### OAuth Consent Screen

Configure it as **External / Testing** with yourself as the only test user. This is fine indefinitely for personal use — Google only requires verification when you intend to publish publicly. For a solo setup, testing mode works forever.

### Scopes: Two-Stage Grant

Two scopes, granted in sequence:

- `https://www.googleapis.com/auth/gmail.modify` — read messages, manage labels, create drafts
- `https://www.googleapis.com/auth/gmail.send` — only added when I promote a category to auto-send

The two-stage scope grant is itself a safety pattern. The drafts phase works without ever granting send access. You literally cannot auto-send until you've separately granted that scope. If the scope list is a security boundary, this is a physical enforcement of it.

### The Refresh Token Flow

Standard OAuth 2.0 Authorization Code, but with a wrinkle: Google's refresh tokens are only issued when you pass `access_type=offline` AND `prompt=consent`. Without those flags you get an access token that expires in an hour and breaks the scheduled task silently. No errors, no alerts — just a task that stops processing.

### The Localhost Callback Script

A ~50-line Node script that opens the browser, captures the auth code on redirect, exchanges it for tokens, and stashes the refresh token in AWS SSM — never touching the repo:

```javascript
// scripts/get-gmail-refresh-token.mjs
import { OAuth2Client } from "google-auth-library";
import http from "http";
import { execSync } from "child_process";

const CLIENT_ID = process.env.GOOGLE_CLIENT_ID;
const CLIENT_SECRET = process.env.GOOGLE_CLIENT_SECRET;
const PORT = 3030;

const oauthClient = new OAuth2Client(
  CLIENT_ID,
  CLIENT_SECRET,
  `http://localhost:${PORT}`,
);

const authUrl = oauthClient.generateAuthUrl({
  access_type: "offline",
  prompt: "consent",
  scope: [
    "https://www.googleapis.com/auth/gmail.modify",
    "https://www.googleapis.com/auth/gmail.send",
  ],
});

const server = http.createServer(async (req, res) => {
  const code = new URL(req.url, `http://localhost:${PORT}`)
    .searchParams.get("code");
  if (!code) return;
  const { tokens } = await oauthClient.getToken(code);
  // Stash refresh_token in SSM SecureString (omitted for brevity)
  res.end("Token saved. Close this window.");
  server.close();
});

server.listen(PORT, () => execSync(`open "${authUrl}"`));
```

Run it once, the refresh token lands in SSM, the deploy pulls it at runtime. The script never runs again unless I want to add a new scope — at which point I edit the scope list and re-run.

## Trigger.dev v4: The Scheduled Task

The scheduled task runs every five minutes via cron:

```typescript
// trigger/gmail-classify.ts
import { schedules, logger } from "@trigger.dev/sdk";

export const gmailClassifyTask = schedules.task({
  id: "gmail-classify",
  cron: "*/5 * * * *",
  maxDuration: 600,
  // googleapis is huge — small-1x (0.5GB) OOMs on cold start before any
  // work runs. medium-1x (2GB) is the smallest preset that gives reliable
  // headroom for googleapis + anthropic + resend in one bundle.
  machine: { preset: "medium-1x" },
  run: async (payload) => {
    const messages = await gmailListLabeled({
      include: "claude/needs-reply",
      exclude: ["claude/drafted", "claude/processing", "claude/error"],
      maxResults: 10,
    });
    for (const ref of messages) {
      try {
        await gmailAddLabel(ref.id, "claude/processing"); // lock
        const message = await gmailGetMessage(ref.id);
        const classification = await classifyAndDraft({ /* ... */ });
        // draft / auto-send / label-swap — see below
      } catch (err) {
        await gmailAddLabel(ref.id, "claude/error");
      }
    }
  },
});
```

**Labels as state.** This is the part that took me a while to appreciate: no database needed, because Gmail labels are persistent and queryable. `claude/needs-reply` triggers, `claude/processing` locks, `claude/drafted` (or `claude/sent`) marks done, `claude/error` flags broken. Re-runs are idempotent because the search query excludes terminal-state labels. Gmail is the state store.

## The syncEnvVars Build Extension

This one cost me about thirty minutes of head-scratching and it's not documented well enough.

`--env-file .env` on the deploy command only loads vars into the CLI process. The actual sync to Trigger.dev's encrypted vault happens via the `syncEnvVars` build extension. Without it, your deployed task has no env vars at runtime even though the deploy claimed success.

```typescript
// trigger.config.ts
import { defineConfig } from "@trigger.dev/sdk";
import { syncEnvVars } from "@trigger.dev/build/extensions/core";

const RUNTIME_ENV_VAR_NAMES = [
  "ANTHROPIC_API_KEY",
  "RESEND_API_KEY",
  "GOOGLE_CLIENT_ID",
  "GOOGLE_CLIENT_SECRET",
  "GOOGLE_REFRESH_TOKEN",
  "AUTO_SEND_CATEGORIES",
] as const;

export default defineConfig({
  project: "proj_...",
  build: {
    extensions: [
      syncEnvVars(async () => {
        const out: Record<string, string> = {};
        for (const name of RUNTIME_ENV_VAR_NAMES) {
          const v = process.env[name];
          if (v) out[name] = v;
        }
        return out;
      }),
    ],
  },
});
```

Your deployed task now has access to those env vars at runtime. Without this extension, it won't — and you won't get an error message explaining why.

## The Anthropic Call with JSON-Schema Enforcement

```typescript
// lib/classify.ts
const response = await client.messages.create({
  model: "claude-opus-4-7",
  max_tokens: 2048,
  system: SYSTEM_PROMPT, // long versioned prompt
  messages: [{
    role: "user",
    content: `INBOUND EMAIL:
From: ${input.from}
Subject: ${input.subject}

${input.body.slice(0, 4000)}

Respond ONLY with the JSON object specified in the system prompt — no preamble, no markdown fences.`,
  }],
});

const text = response.content
  .filter((c) => c.type === "text")
  .map((c) => c.text)
  .join("")
  .trim()
  .replace(/^```(?:json)?\s*/i, "")
  .replace(/\s*```$/i, "");

const parsed = JSON.parse(text);
validateClassification(parsed); // runtime type check
```

**Prompt as code.** The system prompt is a TypeScript string export, not a markdown file. Why? The Trigger.dev bundler doesn't include arbitrary file types in the deploy bundle — `fs.readFileSync` on the `.md` threw `ENOENT` at runtime. Bundling the prompt as a TS const means it's guaranteed to ship with the code. I keep a parallel `.md` version around as human-readable docs, but the TypeScript file is the source of truth at runtime.

The strip-and-parse pattern handles a common problem: Claude sometimes returns the JSON wrapped in code fences and sometimes doesn't. Both are valid — both parse correctly after the regex cleanups.

## Gmail Draft Creation in the Original Thread

```typescript
// lib/gmail.ts
export async function gmailCreateDraft(input) {
  const headerLines = [
    `To: ${input.to}`,
    `Subject: ${input.subject}`,
    "MIME-Version: 1.0",
    "Content-Type: text/plain; charset=UTF-8",
    `In-Reply-To: ${input.inReplyTo}`,
    `References: ${input.references}`,
  ];
  const raw = headerLines.join("\r\n") + "\r\n\r\n" + input.body;
  const rawBase64 = Buffer.from(raw, "utf-8").toString("base64url");
  const res = await gmail.users.drafts.create({
    userId: "me",
    requestBody: {
      message: { threadId: input.threadId, raw: rawBase64 },
    },
  });
  return { id: res.data.id };
}
```

The `threadId` plus In-Reply-To and References headers are what make the draft thread cleanly. When I open Gmail, the draft is attached to the original conversation — no orphan message, no "this doesn't look like a reply" awkwardness. Gmail's conversation view groups it correctly because it's a real reply in a real thread.

## The Auto-Send Opt-In: The Safety Architecture

```typescript
// trigger/gmail-classify.ts (inside the run loop)
const AUTO_SEND_CATEGORIES = new Set(
  (process.env.AUTO_SEND_CATEGORIES ?? "")
    .split(",")
    .map((s) => s.trim().toLowerCase())
    .filter(Boolean),
);

// These two can never auto-send regardless of env, even if listed.
const AUTO_SEND_DISQUALIFIED = new Set([
  "expert-network-followup",
  "noise",
]);

const shouldAutoSend =
  hasReply &&
  AUTO_SEND_CATEGORIES.has(category) &&
  !AUTO_SEND_DISQUALIFIED.has(category);

if (shouldAutoSend) {
  // gmail.users.drafts.send — converts draft into a sent message in-thread
  await gmailSendDraft(draftId);
  await gmailAddLabel(ref.id, "claude/sent"); // distinct from claude/drafted
}
```

This is the keystone. Here's why:

**Per-category graduation.** I don't promote "the whole system to auto-send" — I promote one category at a time, after it's earned the trust. Each category has a track record. I'm not gambling on the system — I'm accumulating evidence.

**Hard-coded disqualifications.** Some categories can never auto-send regardless of what the env var says. Empty-reply categories like `expert-network-followup` — where the action belongs to me (clicking a payment form, signing a W-9) and there's no reply to draft. The hard-coded set is a last line of defense against a config typo promoting a category that shouldn't be promoted.

**Kill switch.** Setting the env var to empty reverts every category back to manual draft mode on the next run. No code change, no panic. One env var edit, one deploy, everything stops.

**Label as audit trail.** Auto-sent messages get `claude/sent`, drafts get `claude/drafted`. Reviewing the audit trail later is a single Gmail filter. I can see every auto-sent message and when it went out.

**Belt notification.** Even when auto-sent, a Resend email lands in my inbox confirming the send. I see every outgoing in real-time and can intervene if something looks wrong.

**Gmail-native threading.** No BCC required. Gmail's conversation view groups sent and received automatically because I'm the one sending — not some shadow copy.

## What I Got Wrong the First Time

Several footguns I hit, in case you're building something similar:

**OOM on the default machine size.** Trigger.dev's default `small-1x` preset (0.5GB) crashed on cold start before any work ran. The `googleapis` library is huge — even with tree-shaking it pulls in code for every Google service. I bumped to `medium-1x` (2GB) and got reliable headroom. The crash was silent in the sense that the cron just stopped advancing; I had to look at the run history in the Trigger.dev dashboard to find the OOM.

**`data` vs `payload` on Resend events.** I sent the trigger event with `{event, email, data: {...}}` and the API returned 202 — looked fine. Except `data` is the wrong field name; Resend's `/events/send` accepts custom payload under `payload`. The 202 was Resend silently dropping my custom data. The catch was when an automation that was supposed to branch on `payload.source` saw nothing. The fix was one field name change; finding it took a code review pass from Cursor Bugbot.

**The bundler doesn't include `.md` files.** I had the system prompt as a versioned markdown file and loaded it with `fs.readFileSync`. The deploy succeeded; the first run threw `ENOENT` at runtime. Trigger.dev's bundler only includes TypeScript / JavaScript imports. The fix: export the prompt as a TypeScript string const.

**`--env-file` doesn't actually push env vars.** I assumed `npx trigger.dev deploy --env-file .env` would upload my env vars to Trigger.dev's vault. It doesn't — that flag only loads them into the deploy CLI process. The actual sync requires the `syncEnvVars` build extension. The task failed at runtime with `GOOGLE_CLIENT_ID env var is required` before I figured this out.

**Quoted values in `.env` and systemd.** A separate service I run loads `.env` via systemd's `EnvironmentFile`. It was silently dropping env vars because the file had single-quoted values (`KEY='value'`). Stripping the quotes brought every secret back. Worth knowing if you're loading `.env` anywhere systemd reads it.
