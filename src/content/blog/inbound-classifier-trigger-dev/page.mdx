## Safety First: The Graduation Ladder

Here's the design constraint I started with: auto-replying to everything is a disaster waiting to happen. The failure mode — confident, context-blind form letters going out under your name — is worse than occasionally spending thirty seconds on a low-stakes reply yourself.

So the system started fully manual. Every classifier output generated a draft Zack reviewed and clicked Send on. That manual phase wasn't a limitation — it was the test bed. Each category accumulated a corpus of confirmed-correct drafts. I was building a track record.

Once a category proved itself — roughly 10 verified-correct outputs in a row, voice rules locked in — it graduated to opt-in auto-send for that specific category. Other categories stayed manual until they earned the promotion. Graduation per-category, not system-wide.

The reversion path is one env var edit and a redeploy. If anything breaks, I flip the switch and everything goes back to manual review on the next run. No code changes, no panic.

## Google Cloud Setup: How the Scheduled Task Gets Inbox Access

Here's the part nobody writes about: how a Trigger.dev task actually gets read access to your Gmail.

### OAuth Client Type: Desktop App

The alternative (Web application) needs a hosted callback URL — something like `https://myapp.ngrok.io/callback`. That's fine if you're running a web app, but my Trigger.dev task is a scheduled background worker. Desktop app clients accept `http://localhost:<port>` as a valid callback, which works from a one-shot local script. Less infra to maintain.

### OAuth Consent Screen

Configure it as **External / Testing** with yourself as the only test user. This is fine indefinitely for personal use — Google only requires verification when you intend to publish publicly. For a solo setup, testing mode works forever.

### Scopes: Two-Stage Grant

Two scopes, granted in sequence:

- `https://www.googleapis.com/auth/gmail.modify` — read messages, manage labels, create drafts
- `https://www.googleapis.com/auth/gmail.send` — only added when we promote a category to auto-send

The two-stage scope grant is itself a safety pattern. Phase 2A (drafts) works without ever granting send access. You literally cannot auto-send until you've separately granted that scope. If the scope list is a security boundary, this is a physical enforcement of it.

### The Refresh Token Flow

Standard OAuth 2.0 Authorization Code, but with a wrinkle: Google's refresh tokens are only issued when you pass `access_type=offline` AND `prompt=consent`. Without those flags you get an access token that expires in an hour and breaks the scheduled task silently. No errors, no alerts — just a task that stops processing.

### The Localhost Callback Script

A 50-line Node script that opens the browser, captures the auth code on redirect, exchanges it for tokens, and stashes the refresh token in AWS SSM — never touching the repo:

```javascript
// scripts/get-gmail-refresh-token.mjs
import { OAuth2Client } from "google-auth-library";
import http from "http";
import { execSync } from "child_process";
import { SSMClient, PutParameterCommand } from "@aws-sdk/client-ssm";

const CLIENT_ID = process.env.GOOGLE_CLIENT_ID;
const CLIENT_SECRET = process.env.GOOGLE_CLIENT_SECRET;
const PORT = 3030;

const oauthClient = new OAuth2Client(CLIENT_ID, CLIENT_SECRET, `http://localhost:${PORT}`);

const authUrl = oauthClient.generateAuthUrl({
  access_type: "offline",
  prompt: "consent",
  scope: [
    "https://www.googleapis.com/auth/gmail.modify",
    "https://www.googleapis.com/auth/gmail.send",
  ],
});

const server = http.createServer((req, res) => {
  const url = new URL(req.url, `http://localhost:${PORT}`);
  if (url.pathname === "/") {
    const code = url.searchParams.get("code");
    if (code) {
      oauthClient.getToken(code).then(async (tokenResponse) => {
        const ssm = new SSMClient({});
        await ssm.send(new PutParameterCommand({
          Name: "/hermes/gmail-refresh-token",
          Value: tokenResponse.tokens.refresh_token,
          Type: "SecureString",
          Overwrite: true,
        }));
        res.end("Token saved to SSM. Close this window.");
        server.close();
      });
    }
  }
});

server.listen(PORT, () => {
  execSync(`open "${authUrl}"`);
});
```

Run it once, the refresh token lands in SSM, the deploy pulls it at runtime. The script never runs again.

## Trigger.dev v4: The Scheduled Task

The scheduled task runs every five minutes via cron:

```typescript
// trigger/gmail-classify.ts
import { schedules, logger } from "@trigger.dev/sdk";

export const gmailClassifyTask = schedules.task({
  id: "gmail-classify",
  cron: "*/5 * * * *",
  maxDuration: 600,
  // googleapis is huge — small-1x (0.5GB) OOMs on cold start before any
  // work runs. medium-1x (2GB) is the smallest preset that gives reliable
  // headroom for googleapis + anthropic + resend in one bundle.
  machine: { preset: "medium-1x" },
  run: async (payload) => {
    const messages = await gmailListLabeled({
      include: "claude/needs-reply",
      exclude: ["claude/drafted", "claude/processing", "claude/error"],
      maxResults: 10,
    });
    for (const ref of messages) {
      try {
        await gmailAddLabel(ref.id, "claude/processing"); // lock
        const message = await gmailGetMessage(ref.id);
        const classification = await classifyAndDraft({ ... });
        // ... draft / auto-send / label-swap (see below)
      } catch (err) {
        await gmailAddLabel(ref.id, "claude/error");
      }
    }
  },
});
```

**Labels as state.** This is the part that took me a while to appreciate: no database needed because Gmail labels are persistent and queryable. `claude/needs-reply` triggers, `claude/processing` locks, `claude/drafted` (or `claude/sent`) marks done, `claude/error` flags broken. Re-runs are idempotent because the search query excludes terminal-state labels. Gmail is the state store.

## The syncEnvVars Build Extension

This one cost me about thirty minutes of head-scratching and it's not documented well enough.

`--env-file .env` on the deploy command only loads vars into the CLI process. The actual sync to Trigger.dev's encrypted vault happens via this build extension. Without it, your deployed task has no env vars at runtime even though the deploy claimed success.

```typescript
// trigger.config.ts
import { defineConfig } from "@trigger.dev/sdk";
import { syncEnvVars } from "@trigger.dev/build/extensions/core";

const RUNTIME_ENV_VAR_NAMES = [
  "ANTHROPIC_API_KEY", "RESEND_API_KEY",
  "GOOGLE_CLIENT_ID", "GOOGLE_CLIENT_SECRET", "GOOGLE_REFRESH_TOKEN",
  "AUTO_SEND_CATEGORIES",
] as const;

export default defineConfig({
  project: "proj_...",
  build: {
    extensions: [
      syncEnvVars(async () => {
        const out: Record<string, string> = {};
        for (const name of RUNTIME_ENV_VAR_NAMES) {
          const v = process.env[name];
          if (v) out[name] = v;
        }
        return out;
      }),
    ],
  },
});
```

Your deployed task now has access to those env vars at runtime. Without this extension, it won't — and you won't get an error message explaining why.

## The Anthropic Call with JSON-Schema Enforcement

```typescript
// lib/classify.ts
const response = await client.messages.create({
  model: "claude-opus-4-7",
  max_tokens: 2048,
  system: SYSTEM_PROMPT, // long versioned prompt
  messages: [{ role: "user", content: `INBOUND EMAIL:
From: ${input.from}
Subject: ${input.subject}

${input.body.slice(0, 4000)}

Respond ONLY with the JSON object specified in the system prompt — no preamble, no markdown fences.` }],
});

const text = response.content
  .filter((c): c is Anthropic.TextBlock => c.type === "text")
  .map((c) => c.text).join("").trim()
  .replace(/^\`\`\`(?:json)?\s*/i, "").replace(/\s*\`\`\`$/i, "");

const parsed = JSON.parse(text) as Classification;
validateClassification(parsed); // runtime type check
```

**Prompt as code.** The system prompt is a TypeScript string export, not a markdown file. Why? The Trigger.dev bundler doesn't include arbitrary file types in the deploy bundle — `fs.readFileSync` on the .md threw ENOENT at runtime. Bundling the prompt as a TS const means it's guaranteed to ship with the code. The .md version stays around as human-readable docs.

The strip-and-parse pattern handles a common problem: Claude sometimes returns the JSON wrapped in ```json fences and sometimes doesn't. Both are valid — both parse correctly after the regex cleanups.

## Gmail Draft Creation in the Original Thread

```typescript
// lib/gmail.ts
export async function gmailCreateDraft(input: DraftInput) {
  const headerLines = [
    `To: ${input.to}`,
    `Subject: ${input.subject}`,
    "MIME-Version: 1.0",
    "Content-Type: text/plain; charset=UTF-8",
    `In-Reply-To: ${input.inReplyTo}`,
    `References: ${input.references}`,
  ];
  const raw = headerLines.join("\r\n") + "\r\n\r\n" + input.body;
  const rawBase64 = Buffer.from(raw, "utf-8").toString("base64url");
  const res = await gmail.users.drafts.create({
    userId: "me",
    requestBody: {
      message: { threadId: input.threadId, raw: rawBase64 },
    },
  });
  return { id: res.data.id! };
}
```

The `threadId` plus In-Reply-To and References headers are what make the draft thread cleanly. When I open Gmail, the draft is attached to the original conversation — no orphan message, no "this doesn't look like a reply" awkwardness. Gmail's conversation view groups it correctly because it's a real reply in a real thread.

## The Auto-Send Opt-In: The Safety Architecture

```typescript
// trigger/gmail-classify.ts (inside the run loop)
const AUTO_SEND_CATEGORIES = new Set(
  (process.env.AUTO_SEND_CATEGORIES ?? "")
    .split(",").map(s => s.trim().toLowerCase()).filter(Boolean),
);

// These two CAN NEVER auto-send regardless of env, even if listed.
const AUTO_SEND_DISQUALIFIED = new Set(["expert-network-followup", "noise"]);

const shouldAutoSend =
  hasReply &&
  AUTO_SEND_CATEGORIES.has(category) &&
  !AUTO_SEND_DISQUALIFIED.has(category);

if (shouldAutoSend) {
  // gmail.users.drafts.send — converts draft into sent message in-thread
  await gmailSendDraft(draftId);
  await gmailAddLabel(ref.id, "claude/sent"); // distinct from claude/drafted
}
```

This is the keystone. Here's why:

**Per-category graduation.** You don't promote "the whole system to auto-send" — you promote one category at a time, after it's earned the trust. Each category has a track record. You're not gambling on the system — you're accumulating evidence.

**Hard-coded disqualifications.** Some categories can never auto-send regardless of what the env var says. Empty-reply categories like `expert-network-followup` where the action belongs to me, not the system. The hard-coded set is the last line of defense.

**Kill switch.** Setting the env to empty reverts every category back to manual draft mode on the next run. No code change, no panic. One env var, one deploy, everything stops.

**Label as audit trail.** Auto-sent messages get `claude/sent`, drafts get `claude/drafted`. Reviewing the audit trail later is a single Gmail filter. I can see every auto-sent message and when it went out.

**Belt notification.** Even when auto-sent, a Resend email lands in my inbox confirming the send. I see every outgoing in real-time and can intervene if something looks wrong.

**Gmail-native threading.** No BCC required. Gmail's conversation view groups sent and received automatically because I'm the one sending — not some shadow copy.

## What I Got Wrong the First Time

**Dry run mode first.** I should have started with a full manual phase and only introduced auto-send after building the calibration corpus. Instead I tried to be clever about thresholds early and spent time unwinding decisions that were premature.

**Prompt versioning from day one.** The current setup has the prompt in an environment variable, which means every change overwrites the previous one. I've lost probably three iterations that were better than what replaced them. Prompt versioning in the repo is the first thing I'd add if I were starting over.

**Recipient count check.** The auto-reply goes out from my personal inbox through Resend's routing. If the classifier fires on a thread with multiple CC'd addresses, the reply goes to everyone. I caught this in testing and added a rule to suppress auto-reply when there are more than two recipients. This is the one that almost shipped.
