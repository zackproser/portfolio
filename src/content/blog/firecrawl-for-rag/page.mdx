## Why Firecrawl Fits RAG

RAG pipelines need clean, structured text from reliable sources. The quality of your retrieval depends heavily on the quality of your source content.

Firecrawl handles the hard part of web scraping for RAG: it navigates JavaScript-heavy pages, extracts clean markdown, and returns content you can directly pipe into your vector database. No BeautifulSoup wrangling. No headless browser management.

## Basic Setup

```python
from firecrawl import FirecrawlApp

app = FirecrawlApp(api_key="your-api-key")

# Scrape a single URL to markdown
result = app.scrape_url(
    "https://example.com/blog/post",
    formats=["markdown"]
)

markdown_content = result.markdown
# Now chunk and embed this for your vector store
```

Firecrawl returns clean markdown that preserves headings, code blocks, and structure. This matters for RAG because document structure helps with both chunking strategies and contextual retrieval.

## Batch Scraping for Knowledge Bases

If you're building a knowledge base from multiple pages:

```python
# Crawl an entire site
crawl_result = app.crawl_url(
    "https://docs.example.com",
    limit=100,
    formats=["markdown"]
)

for page in crawl_result.pages:
    markdown = page.markdown
    url = page.metadata.source

```

The crawler respects robots.txt and can handle pagination, infinite scroll, and JavaScript rendering automatically.

## Handling Dynamic Content

Many modern sites render content client-side. Firecrawl's JavaScript rendering handles this:

```python
result = app.scrape_url(
    "https://react-based-site.com/products",
    formats=["markdown"],
    render_js=True  # Enables JavaScript rendering
)
```

This is where Firecrawl outperforms basic HTTP scrapers. Sites built with React, Vue, or Next.js often return empty responses with conventional tools.

## Chunking Strategies

How you chunk affects retrieval quality. A few approaches:

**By headings**: Use document structure to create semantically coherent chunks. Firecrawl's markdown output preserves heading hierarchy, making this straightforward.

**By token count**: Simple fixed-size chunks (512-1024 tokens) work well as baselines. Overlap between chunks (50-100 tokens) helps maintain context.

**By semantic boundaries**: Split on paragraph breaks or logical sections rather than arbitrary character counts.

## What Firecrawl Doesn't Do

Be clear about scope:

- Firecrawl doesn't handle authentication (login-protected content)
- Rate limits apply based on your plan
- Some sites block scrapers regardless of approach
- You still need to handle deduplication and freshness updates

For production RAG systems that pull from web sources, Firecrawl removes the most painful part of the data ingestion pipeline.
