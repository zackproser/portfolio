## The problem: my AI assistant was blind

I run an always-on AI assistant ([OpenClaw](https://github.com/openclaw/openclaw)) on a System76 Meerkat under my desk. It does morning briefings, monitors my email for time-sensitive opportunities, tracks my Linear tasks, monitors my bank accounts — the whole deal. I wrote about the full setup in [Building an Always-On AI Assistant](/blog/building-always-on-ai-assistant).

The email piece was broken.

Every morning, my assistant runs an email sweep. It was using `gog` — a solid Google Workspace CLI — to search Gmail. The problem: Gmail's API is slow, limited in query power, and paginated in a way that makes bulk analysis basically impossible. In practice, my assistant was seeing the **last 6 emails** and calling it a day. That's not an email sweep. That's peeking through a keyhole.

I need my assistant to instantly query **all** my emails. Cross-reference threads with Linear tickets to understand what's actually done in the world vs. what's still open. Pull up a full conversation thread when an expert network invitation comes in so it has context on whether I've worked with that firm before. Search across 17 years of email history in milliseconds, not minutes.

`gog gmail search` wasn't going to cut it.

## So I had my assistant build the thing

<Image src="https://zackproser.b-cdn.net/images/2026-02-22-21-30-00-build-workflow.webp" alt="Human-AI collaboration loop: chat, code, critique, fix, push" width={800} height={400} />
<figcaption>The build loop: human directs, AI codes, human critiques, AI fixes.</figcaption>

Here's what actually happened: I told Claw (my assistant) what I needed over a couple of chat messages. Claw wrote the initial Go project — a CLI called `gmail-mirror` that syncs your Gmail inbox into a local SQLite database with [FTS5](https://www.sqlite.org/fts5.html) full-text search.

Then I SSH'd into the Meerkat from another terminal and critiqued the code with Claude Code. Found six bugs:

1. **Stalled sync** — a resume token was stuck, so the initial sync never finished
2. **Missing thread command** — the email-check skill was already calling `gmail-mirror thread` but the subcommand didn't exist
3. **Zero dates** — messages where header date parsing failed got stored as `0001-01-01` instead of using Gmail's InternalDate
4. **Single failure kills batch** — one deleted message would abort the entire batch of 50
5. **N+1 query problem** — checking `HasMessage` individually for each of 500 IDs per page instead of batch dedup
6. **Fragile label search** — `LIKE "%INBOX%"` would match any label containing "INBOX" as a substring

Sent the bug report back to Claw. Claw fixed all six, rebuilt, and pushed to a private GitHub repo. Having it in a proper repo means Claw can continue making improvements, fixing bugs, and pushing updates across sessions — the code persists even when the assistant's context resets. Then I had it run the full sync — 205,023 messages, back to 2008. It took about 20 minutes total across a few runs (Gmail rate-limits you at 15,000 requests per minute, so we added exponential backoff and reduced concurrency).

That's the workflow I keep coming back to: **AI builds the first draft, human critiques, AI fixes, repeat.** The whole tool went from "I need this" to "it works" in a single Sunday afternoon.

## How it works

<Image src="https://zackproser.b-cdn.net/images/2026-02-22-20-57-00-gmail-mirror-architecture.webp" alt="gmail-mirror architecture diagram" width={800} height={600} />
<figcaption>Gmail API → gmail-mirror sync → SQLite + FTS5 → CLI search → AI assistant</figcaption>

gmail-mirror is about 940 lines of Go across four files:

- **`cmd/gmail-mirror/main.go`** — CLI built with Cobra. Four commands: `sync`, `search`, `thread`, `stats`.
- **`internal/sync/sync.go`** — Full sync and incremental sync against the Gmail API. Resume tokens, in-memory dedup, concurrent fetchers with exponential backoff.
- **`internal/sync/auth.go`** — Reuses OAuth tokens from the `gog` CLI so you don't need a separate auth flow.
- **`internal/search/search.go`** — FTS5 full-text search with filters for sender, date range, and labels.
- **`internal/store/store.go`** — SQLite schema, FTS5 virtual table, `INSERT OR REPLACE` for idempotent upserts.

### Key decisions

**FTS5 for search.** SQLite's FTS5 extension is absurdly fast for full-text search. A query across 205K messages returns in 20-50ms. Gmail's API takes seconds for the same thing and can't do boolean queries.

**InternalDate over header Date.** Email headers lie. People misconfigure mail servers, forwarding mangles dates, spam has fake timestamps. Gmail's `InternalDate` (when Gmail actually received the message) is always correct. Use it as primary, fall back to header date only if InternalDate is missing.

**In-memory dedup.** During a full sync, you're paging through every message ID in your mailbox. The original code did a `SELECT COUNT(*) FROM messages WHERE message_id = ?` for each of the 500 IDs per page. That's 500 round-trips per page across potentially thousands of pages. The fix: load all existing message IDs into a Go `map[string]struct{}` at the start. One query, O(1) lookups after that.

```go
existingIDs, err := s.store.AllMessageIDs()
// ... loaded 205,023 IDs in 160ms

for _, m := range resp.Messages {
    if _, exists := existingIDs[m.Id]; exists {
        continue // skip, already have it
    }
    batch = append(batch, m.Id)
}
```

**Exponential backoff for rate limits.** Gmail allows 15,000 requests per minute. At 5 concurrent fetchers doing ~25 messages/second, you'll bump into this during a full sync. Instead of failing the batch, retry with 1s, 2s, 4s, 8s, 16s backoff:

```go
for attempt := 0; attempt < 5; attempt++ {
    msg, err = s.svc.Users.Messages.Get("me", msgID).Format("full").Do()
    if err == nil {
        break
    }
    if strings.Contains(err.Error(), "rateLimitExceeded") {
        backoff := time.Duration(1<<uint(attempt)) * time.Second
        time.Sleep(backoff)
        continue
    }
    // Non-rate-limit error — log and skip
    break
}
```

**Resume tokens for interrupted syncs.** A full sync of 205K messages takes ~20 minutes. If it gets interrupted (process killed, network hiccup, laptop goes to sleep), the resume token lets you pick up exactly where you left off. No re-scanning, no wasted work.

**`json_each()` for label search.** Labels are stored as a JSON array. The original code used `LIKE "%INBOX%"` which would incorrectly match a hypothetical label called `NOT_INBOX`. Using SQLite's `json_each()` function gives exact matching:

```go
conditions = append(conditions, 
    "EXISTS (SELECT 1 FROM json_each(m.labels) WHERE json_each.value = ?)")
```

### The daily workflow

Every morning at 8:30 AM, my assistant:

1. Runs `gmail-mirror sync` to pull the latest messages (takes ~2 seconds for incremental sync)
2. Searches for urgent/time-sensitive emails using FTS5 queries
3. Cross-references with Linear to see what's actually in progress vs. stale
4. Flags expert network invitations for immediate response
5. Summarizes everything in a morning briefing delivered to Discord

The difference from before: instead of seeing 6 recent emails through a Gmail API keyhole, it's searching across 205,023 messages with full context. Thread history, sender patterns, date-based analysis — all in milliseconds.

## Performance

```
Total messages: 205,023
Date range:     2008 to 2026 (17 years)
Database size:  ~2 GB
Sync time:      ~2 seconds (incremental), ~20 minutes (full)
Search latency: 20-50ms
```

| Query | Gmail Web | Gmail API | gmail-mirror |
|-------|-----------|-----------|--------------|
| Simple keyword | 2-5s | 1-3s | **20-50ms** |
| Boolean + date range | Unreliable | Slow pagination | **30-80ms** |
| Bulk analysis (200+ results) | Impossible | Minutes | **< 1 second** |
