## The Problem: Gmail Search Sucks for Power Users

You know that feeling when you *know* there's an important email buried somewhere in your inbox, but Gmail's search just... can't find it? Or when you want to do analysis across years of email history, but the Gmail web interface times out after scrolling through a few hundred messages?

I've been using the same Gmail account since 2008. That's **17 years** and **205,023 messages** of digital archaeology just sitting there, mostly inaccessible. Gmail's search API is slow, limited in query power, and hopeless for bulk analysis.

So I built **gmail-mirror** — a Go CLI tool that syncs your entire Gmail inbox into a local SQLite database with FTS5 full-text search. Queries that take Gmail 10+ seconds now return in milliseconds. Complex filters that are impossible in Gmail's web UI work perfectly.

And then I pointed it at my inbox to see what I'd been missing.

### Key Technical Decisions

**1. Reuse gog CLI OAuth Tokens**
Instead of implementing yet another OAuth flow, gmail-mirror piggybacks on the excellent [gog CLI](https://github.com/hduplooy/gog) for Google Workspace access:

```go
// internal/sync/auth.go - Token reuse from gog CLI
func LoadTokenFromFile(tokenPath string) (*oauth2.Token, error) {
    data, err := os.ReadFile(tokenPath)
    if err != nil {
        return nil, fmt.Errorf("read token file: %w", err)
    }
    
    var token oauth2.Token
    if err := json.Unmarshal(data, &token); err != nil {
        return nil, fmt.Errorf("unmarshal token: %w", err)
    }
    
    return &token, nil
}
```

**2. FTS5 for Instant Full-Text Search**
SQLite's FTS5 extension provides Google-quality search performance locally:

```go
// internal/store/store.go - FTS5 virtual table
const createFTSTableSQL = `
CREATE VIRTUAL TABLE IF NOT EXISTS messages_fts USING fts5(
    subject, 
    from_addr, 
    to_addr, 
    body_text, 
    content='messages', 
    content_rowid='rowid'
);`
```

Search queries compile to optimized FTS5 SQL:

```go
// internal/search/search.go - FTS5 query construction
if opts.Query != "" {
    conditions = append(conditions, "m.rowid IN (SELECT rowid FROM messages_fts WHERE messages_fts MATCH ?)")
    args = append(args, opts.Query)
}
```

**3. Resume Tokens for Interrupted Syncs**
Full syncs can take hours. Resume tokens ensure you never lose progress:

```go
// internal/sync/sync.go - Resume token handling
func (s *Syncer) FullSync(ctx context.Context) error {
    // Check for resume point
    resumeToken, _ := s.store.GetSyncState("resume_page_token")
    syncedCount, _ := s.store.MessageCount()
    if resumeToken != "" {
        s.log.Info("resuming sync", "already_synced", syncedCount)
    }
    
    // ... sync logic with periodic token saves
    if resp.NextPageToken != "" {
        s.store.SetSyncState("resume_page_token", resp.NextPageToken)
    }
}
```

**4. In-Memory Dedup for Performance**
Loading all existing message IDs into memory avoids O(n) database queries during sync:

```go
// Load all existing message IDs for fast dedup (one query, not 500 per page)
existingIDs, err := s.store.AllMessageIDs()
if err != nil {
    return fmt.Errorf("load existing IDs: %w", err)
}

// Skip if already in DB (checked via in-memory set, not per-message query)
if _, exists := existingIDs[m.Id]; exists {
    continue
}
```

**5. Rate Limit Retry with Exponential Backoff**
Gmail API has quotas. Handle them gracefully:

```go
// internal/sync/sync.go - Exponential backoff retry
func (s *Syncer) fetchWithRetry(ctx context.Context, messageID string, attempt int) (*gmail.Message, error) {
    if attempt > 5 {
        return nil, fmt.Errorf("max retries exceeded for message %s", messageID)
    }
    
    msg, err := s.svc.Users.Messages.Get("me", messageID).Do()
    if err != nil && strings.Contains(err.Error(), "quotaExceeded") {
        delay := time.Duration(1<<attempt) * time.Second
        s.log.Warn("quota exceeded, retrying", "delay", delay, "attempt", attempt)
        time.Sleep(delay)
        return s.fetchWithRetry(ctx, messageID, attempt+1)
    }
    
    return msg, err
}
```

### Incremental Sync Strategy

Full syncs take hours. Incremental syncs take seconds:

```go
func (s *Syncer) IncrementalSync(ctx context.Context) error {
    lastHistoryID, _ := s.store.GetSyncState("last_history_id")
    if lastHistoryID == "" {
        return fmt.Errorf("no history ID found, run full sync first")
    }
    
    historyCall := s.svc.Users.History.List("me").StartHistoryId(lastHistoryID)
    resp, err := historyCall.Do()
    if err != nil {
        return fmt.Errorf("list history: %w", err)
    }
    
    // Process only new/modified messages
    for _, history := range resp.History {
        for _, msg := range history.MessagesAdded {
            // Fetch and store new message
        }
    }
}
```

Gmail's History API tracks changes since your last sync. Perfect for keeping local mirrors fresh.

### Concurrent Processing

Batch fetching with controlled concurrency prevents API rate limits:

```go
func (s *Syncer) fetchAndStore(ctx context.Context, messageIDs []string) error {
    const maxWorkers = 5
    sem := make(chan struct{}, maxWorkers)
    var wg sync.WaitGroup
    
    for _, id := range messageIDs {
        sem <- struct{}{}
        wg.Add(1)
        
        go func(messageID string) {
            defer wg.Done()
            defer func() { <-sem }()
            
            msg, err := s.fetchWithRetry(ctx, messageID, 0)
            if err != nil {
                s.log.Error("fetch message failed", "id", messageID, "error", err)
                return
            }
            
            if err := s.store.InsertMessage(msg); err != nil {
                s.log.Error("store message failed", "id", messageID, "error", err)
            }
        }(id)
    }
    
    wg.Wait()
    return nil
}
```
