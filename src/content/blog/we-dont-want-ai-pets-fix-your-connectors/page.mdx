# We don't want AI pets. Fix your fucking connectors.

This started as a Slack message. A colleague asked who thought the Codex pets thing was a good idea — "these people have so much time and tokens on their hands." My answer: when the connector I pay for is broken, a virtual pet advertises where the company spent its attention.

Open-weight models keep proving how capable they are. Developers will keep building local inference pools around them. The frontier labs still have a moat: the product experience around the model. Every connector that lies about its state punches another hole in it. When the paid, hosted experience breaks on basic work, "just run it yourself" becomes a rational response.

So here is the hate mail from the edge of applied AI, where these products meet real work.

## The pets

In May, OpenAI added [AI-generated pets to Codex](https://tech.yahoo.com/ai/chatgpt/articles/openai-introduces-ai-generated-pets-193153770.html): animated companions that float over your editor and report what the agent is doing. They do no coding. OpenAI also offered thirty days of ChatGPT Pro to the creators of ten favorite community pets. The pets come with a subscription funnel.

I can enjoy a Clippy homage. Delight belongs in a finished product. A broken connector sitting beside it turns delight into evidence about priorities.

## Exhibit A: the ChatGPT Work Google Drive connector doesn't work

I tested this today on ChatGPT Work. I connected the Google Drive connector, reloaded the conversation, reconnected, and reloaded again. The tool still returned that it had been disabled.

<Image src="https://zackproser.b-cdn.net/images/chatgpt-drive-disabled.webp" alt="A ChatGPT conversation: after reconnecting the Google Drive connector and reloading multiple times, the assistant confirms it invoked the connector directly and it returned 'The Google_Drive tool has been disabled' — concluding the connector is failing on the platform side, not a permissions or folder-link problem" width={1785} height={636} />

My permissions and folder link were fine. Reloading did nothing. The connector said it was connected while failing on the platform side inside the conversation.

The OpenAI community forums show the same failure: the connector displays *Connected*, then returns "no access" or greys out in the composer with a disabled icon and no explanation. [Suddenly disabled, still shows connected, returns "No Access."](https://community.openai.com/t/google-drive-connector-suddenly-disabled-still-shows-as-connected-but-returns-no-access/1361965) [Connected but ChatGPT sees no files.](https://community.openai.com/t/googledrive-connector-is-connected-but-chatgpt-does-not-see-anyfiles/1375549) [Shows connected but can't be selected in chat.](https://community.openai.com/t/google-drive-connector-shows-as-connected-but-cant-be-selected-in-chat-interface/1247537)

A connector that lies about its state burns my time proving the vendor's bug for them. It also makes a local model look better. Offline software never claims it connected to Drive and then quietly disables the tool.

## Exhibit B: Claude's GitHub integration can't hold two orgs in its head

I hit this one last week. The Claude GitHub integration — the same plumbing behind the Claude tag and Claude Code on the web — uses the GitHub org settings wired into the admin panel. Connect a personal GitHub org beside an existing org and it fails silently. The repos never appear.

This exact failure is documented and open in Anthropic's own tracker.

<Image src="https://zackproser.b-cdn.net/images/issue-personal-repos.webp" alt="Open GitHub issue #57161 on anthropics/claude-code: 'Personal GitHub repos not showing in Claude Code web repo picker despite app installed and connector connected.' The issue is labeled duplicate and open; the body states repos show 'No repos match' even though the GitHub App is installed and the connector shows Connected" width={1000} height={900} />

Note the label: **duplicate**. Triage already recognizes repeat reports of the bug. [#57161](https://github.com/anthropics/claude-code/issues/57161) sits beside [#18467](https://github.com/anthropics/claude-code/issues/18467) (organization repos work while personal repos stay invisible), [#68517](https://github.com/anthropics/claude-code/issues/68517) (full access granted, repo still unreadable because of backend sync), [#32479](https://github.com/anthropics/claude-code/issues/32479) (connected and unrecognized), and [#64130](https://github.com/anthropics/claude-code/issues/64130) (the OAuth-only flow leaves private repos inaccessible to remote agents).

The `anthropics/claude-code` repository has about 11,000 open issues, with roughly 2,500 touching GitHub integration. The repository covers the CLI, web app, and GitHub app together, so those issues cover more than this bug. The volume still measures the connector failure class in the thousands. Anthropic's own [help article](https://support.claude.com/en/articles/10167454-use-the-github-integration) documents the SSO and organization-authorization footgun waiting for anyone with more than one org.

The Claude tag draws on separate usage credits and is one of the most expensive ways to run AI right now. Meanwhile, open-weight models keep approaching the frontier on benchmarks and proving capable in developers' hands. A premium earns its price through the whole product experience. A GitHub integration that cannot attach a second org throws that advantage away.

## The actual complaint

I like these companies. I use both every day. I run a fleet of agents on this stack. I care enough to be angry when the expensive path loses to the simple one on reliability.

Connectors are where the model touches my real work: Drive, repos, systems of record. Their reliability is the product. A flaky connector poisons everything above it, wastes a Tuesday afternoon, and sends developers toward inference they control.

I know different teams build pets and connectors. From my seat outside the org, that is an explanation of the org chart, not a defense of the product. Prioritization is an org-level signal. You can make it rain tamagotchis once the basic connectors are rock solid and GitHub integration is bulletproof. Until then, you are doing something wrong.

My new MacBook arrives tomorrow. I'm excited to bring up llama.cpp, switch the wifi radio off entirely, run open-weight models, and see what damage I can do. No hosted workflow. No connector state to distrust. The exit ramp exists, and every broken integration makes me more interested in taking it.

Make the connectors tell the truth. Let me attach two orgs. Return a real error when something fails. Then make it rain tamagotchis. Until then, fix your fucking connectors.
