Scraping a JavaScript-rendered page at the wrong moment returns garbage: loading spinners, empty lists, skeleton screens. The page isn't empty — it's just not done yet. The data is coming. Your scraper arrived too early.

Timing is the core problem in modern web scraping. JavaScript frameworks give you content on a schedule you don't control. Your job is to detect when the page is ready, not assume it is.

## Why Timing Matters

A modern web page goes through multiple states between the initial HTTP response and the final rendered output:

1. **HTML received.** The browser has the HTML shell. JavaScript files are referenced but not yet downloaded.
2. **JavaScript parsing.** The browser parses and executes JavaScript bundles. Framework code initializes.
3. **Component mount.** React/Vue/Angular components mount to the DOM with initial (often empty) state.
4. **Data fetching.** Components fire API requests. Network round trips happen.
5. **Re-render.** Components receive data, update state, and re-render with real content.
6. **Stable DOM.** No more pending mutations. The page is ready to scrape.

A scraper that fires at step 3 sees empty component shells. One that fires at step 4 sees loading states. Only step 6 gives you the complete picture.

The challenge: step 6 is not a specific event. It's a condition that emerges when all async operations settle. Different pages have different settlement times depending on how many API calls they make, how fast those APIs respond, and how complex their rendering logic is.

## The `waitFor` Parameter

<AffiliateLink product="firecrawl" campaign={campaign}>Firecrawl</AffiliateLink>'s most direct timing control is `waitFor` — a millisecond delay applied after the initial page load event:

```javascript
import Firecrawl from '@mendable/firecrawl-js'

const app = new Firecrawl({ apiKey: process.env.FIRECRAWL_API_KEY })

// Simple wait: pause 2 seconds after page load before extracting
const result = await app.scrapeUrl('https://react-app.com/dashboard', {
  formats: ['markdown'],
  waitFor: 2000,
})
```

`waitFor` is a blunt instrument but often the right one. It works well when:
- The page has a predictable load time
- You're scraping at low frequency and latency doesn't matter much
- You don't know which specific elements to wait for

The downside: you're either waiting too long (wasting time) or not long enough (getting incomplete content). For high-volume scraping, calibrate `waitFor` to the minimum that reliably gets complete content.

## Waiting for Specific Elements

A smarter approach than fixed delays: wait for a specific DOM element that only appears once the page has fully rendered. Most well-designed apps have these — a main content container, a data grid, a specific heading that's data-driven:

```javascript
// Wait for a specific element that confirms the page is ready
const result = await app.scrapeUrl('https://angular-app.com/reports', {
  formats: ['markdown'],
  actions: [
    // Wait up to 10 seconds for the reports table to appear
    { type: 'waitForSelector', selector: '.reports-table tbody tr', timeout: 10000 },
    // Small additional buffer for any post-render effects
    { type: 'wait', milliseconds: 500 },
  ],
  waitFor: 1000,
})
```

`waitForSelector` is more reliable than `waitFor` for these reasons:
- It adapts to actual page behavior instead of assuming a fixed time
- Fast pages return immediately instead of waiting the full duration
- Slow pages (API timeouts, heavy computations) get the time they need

Good selectors to wait for:
- The main data container class (`.product-list`, `.data-table`, `[data-loaded="true"]`)
- A specific heading that's rendered from data, not static HTML
- The count/pagination element that appears after data loads (`Showing 1-20 of 847`)

## Waiting for Loading States to Disappear

Inverse selection — waiting for loading indicators to go away — is equally useful:

```javascript
// Wait for loading spinner to disappear before scraping
const result = await app.scrapeUrl('https://vue-dashboard.com/analytics', {
  formats: ['markdown'],
  actions: [
    // Wait for spinner to disappear (means data has loaded)
    { type: 'waitForSelector', selector: '.loading-spinner', state: 'hidden', timeout: 15000 },
    { type: 'waitForSelector', selector: '.skeleton-loader', state: 'hidden', timeout: 10000 },
    { type: 'wait', milliseconds: 500 }, // Brief stability buffer
  ],
  waitFor: 1000,
})
```

Common loading indicators to watch for:
- `.loading`, `.spinner`, `.skeleton`, `[aria-busy="true"]`
- Framework-specific: Angular's `mat-progress-spinner`, React Query's `isLoading` classes
- Custom data attributes: `[data-loading="true"]`, `[data-state="loading"]`

## Handling Network-Dependent Rendering

Some pages make multiple sequential API calls before rendering fully. Each call adds latency:

```javascript
// Page that makes multiple API calls before rendering complete content
const result = await app.scrapeUrl('https://financial-dashboard.com/portfolio', {
  formats: ['markdown'],
  actions: [
    // Wait for portfolio summary (first API call)
    { type: 'waitForSelector', selector: '.portfolio-total', timeout: 8000 },
    // Wait for holdings list (second API call)
    { type: 'waitForSelector', selector: '.holdings-list', timeout: 8000 },
    // Wait for performance chart (third API call, might be slower)
    { type: 'waitForSelector', selector: '.performance-chart canvas', timeout: 12000 },
    { type: 'wait', milliseconds: 1000 },
  ],
  waitFor: 2000,
})
```

## Diagnosing Render Timing Problems

If you're getting incomplete content, these debugging approaches help identify the timing issue:

**Increase `waitFor` dramatically and see if results improve.** Set it to 10 seconds. If you get complete content, the issue is purely timing. Then dial it back to find the minimum that works.

**Check what you do get back.** If the scraped content shows "Loading...", skeleton text, or error states, you know exactly what render stage the scraper caught. Look for the selector that corresponds to the final loaded state.

**Compare with raw HTML.** Scrape with `formats: ['html']` and look at the raw markup. Is the content present but your extraction logic is broken? Or is the content genuinely absent (rendering problem)?

```javascript
// Debugging: capture HTML at different wait times
const shortWait = await app.scrapeUrl('https://problem-site.com', {
  formats: ['html'],
  waitFor: 500,
})

const longWait = await app.scrapeUrl('https://problem-site.com', {
  formats: ['html'],
  waitFor: 5000,
})

// Compare — if longWait has more content, it's a timing issue
console.log('Short wait HTML length:', shortWait.html?.length)
console.log('Long wait HTML length:', longWait.html?.length)
```

## Server-Side Rendering as the Ideal Target

If you can identify that a page uses SSR (Next.js, Nuxt, Angular Universal, Remix), it's the best-case scenario for scraping. SSR pages send real content in the initial HTML response. The timing problem is reduced to just hydration effects, which are generally fast:

```javascript
// SSR pages need minimal wait — content is in initial HTML
const ssrPage = await app.scrapeUrl('https://nextjs-blog.com/posts/my-post', {
  formats: ['markdown'],
  waitFor: 500, // Just enough for hydration effects
})
```

Getting timing right is the difference between reliable production scrapers and brittle ones that fail unpredictably. The tools are there — `waitFor` for simple cases, selector-based waiting for precision, and incremental debugging when results don't match expectations.
