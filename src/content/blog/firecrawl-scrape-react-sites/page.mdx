You fire up `curl` on a React site and get back a shell: `<div id="root"></div>`. That's the entire page. The actual content — product listings, article bodies, user profiles — lives in JavaScript bundles that execute in the browser, not in the HTML the server sends.

Traditional scrapers are blind to this. They see the empty shell and return nothing useful. Scraping React sites requires a tool that runs the JavaScript first and gives you the fully-rendered DOM.

## Why React Sites Break Standard Scrapers

React applications work differently from classic server-rendered sites:

**Client-side rendering (CSR).** With pure CSR, the server sends a minimal HTML document and a JavaScript bundle. React mounts to the DOM, fetches data from APIs, and renders components in the browser. There's nothing to scrape until that process completes.

**Next.js hybrid rendering.** Next.js complicates things further. A single Next.js app might use static generation for blog posts, server-side rendering for product pages, and client-side rendering for user dashboards. Each page has different scraping characteristics.

**Hydration timing.** Even server-rendered React sends a "hydration" step where the client JavaScript takes over. Interactive elements, dynamic data, and component state aren't visible until hydration finishes.

**Route-based code splitting.** React apps split JavaScript by route. Navigating between pages triggers new bundle downloads and render cycles. A scraper that doesn't simulate navigation misses content on sub-pages.

## Scraping React with Firecrawl

<AffiliateLink product="firecrawl" campaign={campaign}>Firecrawl</AffiliateLink> runs a full headless browser under the hood. It loads the page, executes JavaScript, waits for the DOM to stabilize, and then extracts content from the rendered output — not the raw HTML.

```javascript
import Firecrawl from '@mendable/firecrawl-js'

const app = new Firecrawl({ apiKey: process.env.FIRECRAWL_API_KEY })

// Scrape a React SPA — Firecrawl handles the JS execution
const result = await app.scrapeUrl('https://react-ecommerce-site.com/products', {
  formats: ['markdown', 'html'],
  waitFor: 2000, // ms to wait after page load for React to finish rendering
})

console.log(result.markdown) // Full product listing, not an empty div
```

The `waitFor` option is key for React apps. It tells Firecrawl to pause after the initial page load and give React time to complete its render cycle before extracting content.

## Handling Next.js Applications

Next.js sites require some additional considerations depending on the rendering strategy:

**Static Generation (SSG).** Pages pre-rendered at build time behave like standard HTML pages. Firecrawl handles these with no special configuration — the content is already in the initial response.

**Server-Side Rendering (SSR).** SSR pages render on the server per request and arrive mostly complete. Still use `waitFor` for client-side hydration effects that might modify the DOM after load.

**Incremental Static Regeneration (ISR).** ISR pages are cached static files that regenerate in the background. They behave like SSG for scraping purposes.

**Client-side data fetching.** Components that fetch data via `useEffect` or SWR hooks load after the initial render. These require longer wait times:

```javascript
// For Next.js pages with heavy client-side data fetching
const result = await app.scrapeUrl('https://nextjs-dashboard.com/analytics', {
  formats: ['markdown'],
  waitFor: 3000, // Give time for useEffect data fetches to complete
  actions: [
    { type: 'wait', milliseconds: 1000 }, // Additional stability wait
  ]
})
```

## Crawling Multi-Page React Apps

React Router and Next.js routing handle navigation in JavaScript, not through standard HTTP redirects. Firecrawl's crawler understands this and follows in-app navigation links correctly:

```javascript
// Crawl a full React app, following client-side routes
const crawlResult = await app.crawlUrl('https://react-app.com', {
  limit: 100,
  scrapeOptions: {
    formats: ['markdown'],
    waitFor: 2000,
  },
  // Firecrawl follows <a> tags including those rendered by React Router
  allowBackwardLinks: false,
})

// Process all scraped pages
for (const page of crawlResult.data) {
  console.log(`${page.metadata?.sourceURL}: ${page.markdown?.slice(0, 200)}`)
}
```

## Extracting Structured Data from React Components

React apps often organize information in component hierarchies. Firecrawl's structured extraction lets you target specific data patterns regardless of the underlying component structure:

```javascript
import { z } from 'zod'

// Define the shape of the data you want
const ProductSchema = z.object({
  products: z.array(z.object({
    name: z.string(),
    price: z.string(),
    rating: z.string().optional(),
    description: z.string().optional(),
  }))
})

// Extract structured data from a React product listing
const result = await app.scrapeUrl('https://react-shop.com/products', {
  formats: ['extract'],
  extract: {
    schema: ProductSchema,
    prompt: 'Extract all product listings with their names, prices, and ratings'
  },
  waitFor: 2000,
})

console.log(result.extract.products)
// [{name: 'Widget Pro', price: '$29.99', rating: '4.5', ...}, ...]
```

## Dealing with Authentication and Protected Routes

Many React apps gate content behind login. Firecrawl supports custom headers and cookie injection for authenticated scraping:

```javascript
// Scrape a React app that requires authentication
const result = await app.scrapeUrl('https://app.example.com/dashboard', {
  formats: ['markdown'],
  waitFor: 3000,
  headers: {
    'Authorization': `Bearer ${process.env.APP_TOKEN}`,
  },
  // Or use cookies for session-based auth
  // cookies: [{ name: 'session', value: 'your-session-token', domain: 'app.example.com' }]
})
```

## Performance Considerations

React app scraping is slower than static HTML scraping because of the JavaScript execution overhead. A few strategies help:

**Batch requests.** Instead of scraping pages sequentially, use Firecrawl's crawl API to process multiple pages in parallel. The service manages concurrency and rate limiting automatically.

**Target specific routes.** If you only need data from certain sections of a React app, provide `includePaths` to limit crawl scope and skip unneeded routes.

**Cache results aggressively.** React app content often changes less frequently than you might expect. Cache scraped results and set re-scrape intervals based on actual content update frequency, not worst-case assumptions.

**Use structured extraction.** LLM-based extraction (Firecrawl's `extract` format) processes the rendered markdown into structured JSON in one pass, eliminating a separate parsing step.

React sites are the dominant pattern for modern web apps. Getting scraping right means understanding the rendering lifecycle, using the right wait times, and choosing a tool that treats JavaScript execution as a first-class requirement rather than an afterthought.
