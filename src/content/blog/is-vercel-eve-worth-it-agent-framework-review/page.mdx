# Vercel's eve agentic framework review. Is eve worth it?

<Image src="https://zackproser.b-cdn.net/images/is-eve-worth-it-hero.jpg" alt="Pixel art of a developer at a fork in the road holding a signpost, weighing a clean pre-built agent factory against a workshop of hand-wired glue code, a glowing question mark overhead" width={1408} height={792} />

I spent the last few weeks building a real business on Vercel's [eve](https://vercel.com/eve): [a fleet of agents that run my website](/blog/reviewing-vercels-eve-agent-framework), deployed, handing work to each other over authenticated POSTs and parking for my approval in Slack. That post is the long build story. After reading it, people ask for the verdict.

For me, the verdict is an emphatic yes, for the way I build. I reached for eve because I had built the hard version of this exact system somewhere else, watched it strain under its own growth, and learned the lesson eve bakes into its design. The useful answer to "is eve worth it" starts with why I reached for it, what it fixed, and where its edges still cut.

## What eve is, in one paragraph

<Image src="https://zackproser.b-cdn.net/images/is-eve-worth-it-directory-model.jpg" alt="Pixel art of an agent depicted as an open directory of labeled folders — instructions.md, tools, skills, channels, schedules — with a robot assembling itself from the files, on a foundation slab of durability, sandbox, approvals, and connectors" width={1408} height={792} />

eve is an open-source agent framework from Vercel for building, running, and scaling agents in production, with durable execution, sandboxed compute, approvals, channels, tracing, and evals built in. The core idea is that an agent is a *directory of files*: instructions in markdown, tools as one TypeScript file each (the filename becomes the tool name, no registration), skills as markdown playbooks loaded on demand, plus channels and schedules as files. eve discovers that directory and compiles it into an app that runs on Vercel Functions ([the docs](https://vercel.com/docs/eve) and [source](https://github.com/vercel/eve) are both public). Vercel pitched it on stage as "Next.js for agents," and that comparison is fair: the same convention-over-configuration bet Next.js made for the web, aimed at agents. The whole shape is a directory you can read in one glance:

```text
website-manager/
└── agent/
    ├── instructions.md   # the always-on system prompt
    ├── agent.ts          # model + runtime config (optional)
    ├── tools/            # one typed tool per file (filename = tool name)
    ├── skills/           # markdown procedures loaded on demand
    ├── channels/         # Slack, HTTP, web, etc.
    └── schedules/        # cron jobs as files
```

The useful part is what that bundle saves me from writing. Production agents need durable sessions that survive crashes and deploys, sandboxed compute, human-in-the-loop approvals, channel connectors, and cron. eve ships that plumbing inside the framework, so I do not write and maintain the glue. Work that took days to scaffold takes minutes. That pitch held through most of my build.

## Why I reached for it: the lesson from the last build

Before the portfolio fleet, I built [the WorkOS blog bot](https://workos.com/blog/cloudflare-workers-workflows-ai-blog-bot): a Slack-native publishing system on Cloudflare Workers and Durable Workflows. It was a capable distributed system hiding behind a Slack message: a multi-model pipeline (Haiku to classify, Opus to draft), durable step functions with per-step retries and checkpointing, D1 for state, R2 for artifacts, seven external APIs, a proposal system that pitched its own posts. I'm proud of it. I also felt the places where it started to hurt.

The lesson from that build explains why eve landed for me: **as an agentic system gains capabilities, organization and separation of concerns have to become design constraints, or the next feature can regress existing behavior.** Each new capability I bolted on shared more surface with the ones in place: the same router, the same state machine, the same tangle of connector code. Past a certain complexity, a change meant to add one behavior would degrade another one I'd shipped, because the concerns lacked the clean separation I needed to change one part without touching the rest. The plumbing and the judgment lived too close together.

That failure mode is the one eve is built to prevent, which makes "is it worth it" a concrete question for me. eve makes an agent a *directory*: one job, its own instructions, its own typed tools, its own scoped secrets. Vercel owns the shared substrate (durability, sandboxes, connectors, approvals, cron), so I avoid copying that substrate across each bot and letting it drift. The organization I had to impose by hand and defend by discipline on the Cloudflare build is the framework's default. After I split the portfolio work into separate single-purpose bots on eve, adding a capability to one no longer regressed another, because they shared the framework underneath them and little else. For me, the value came from a pain in the last system that this one removed.

## What it nails

After building three separate bots on it (a content bot, an ops bot, and a growth bot), I call these genuine strengths.

**Conventions kill boilerplate.** A bot is an `agent/` directory and the filename is the wiring. eve removes registries, registration calls, and framework ceremony from the path between "I wrote a tool" and "the model can call it." This is the biggest day-to-day win. It helps when a coding agent is building the bot, because the model has few places to get the wiring wrong.

**It absorbs glue-code sprawl.** The failure mode of hand-rolling agents is repeated connector code for Slack, cron, durable sessions, and secrets. That glue is where the bugs live. eve makes that its problem. My three repos hold the code that differs between the bots. The connector code does not get duplicated three times and rot in three different ways, because I did not write it.

**Durability and human-in-the-loop are first-class.** Sessions run on Vercel's durable workflow machinery: state persists between steps, so a task survives a deploy or a cold start and resumes where it stopped. And a `needsApproval: true` on any tool parks the session until a human responds. Those are the two parts I would otherwise bolt on with less care, and having them baked in is why the fleet is safe to leave running.

**Vercel-native deploy.** An eve agent is an ordinary Vercel project, so `vercel deploy` ships the same code from your local run. I did not need an agent-specific build target or separate pipeline.

**Slack Connect, once it's wired, is the best I've used.** You attach a managed connector instead of babysitting a webhook bridge you maintain yourself. That's what turned my experiment into something I use from my phone each day.

## Where I lost time

<Image src="https://zackproser.b-cdn.net/images/is-eve-worth-it-friction.jpg" alt="Pixel art of the sharp edges of a beta tool — a silent-delivery pipe with no error light, a spinning beta-version-drift dial, a stack of duplicate app icons — with a developer holding a 'pin your versions' shield" width={1408} height={792} />

eve is in public beta, and it reads like it. The conventions are excellent; most friction came from beta churn and connector setup. Budget real time for these.

**The silent trigger-path gotcha.** Wiring Slack through Vercel Connect, the event trigger has to point at the exact route eve serves (`/eve/v1/slack`, set with `--trigger-path`). Miss the flag and Connect registers its webhook at a default path, then forwards verified Slack events to a route eve does not listen on. I saw no 404, no failed-delivery banner, and the Agent Runs page stayed empty because no run started. The result was silence. This one cost me real debugging time, because a bot that goes silent with no error sends you hunting through your own code before you check the wiring.

**Beta dependency drift.** A clean install can float pre-release builds of the underlying packages: in my build a CANARY `@ai-sdk` build broke a tool loop mid-run with a type-validation error, and a too-new `@vercel/connect` shipped a verifier that 401'd Slack events. Pin `@ai-sdk`, `@vercel/connect`, and `eve` itself, and commit the lockfile. The API will move under you before general availability. That's the deal with a beta, and eve's docs say as much.

**Thin observability when delivery fails with no signal.** eve gives you an Agent Runs view for sessions, turns, and token usage, which is good. The failures that hurt in beta happen before a run starts, when an event fails to arrive. The run view cannot show those, because no run exists. The work became connector diffing because no stack trace pointed at the problem.

**Platform-shaped limits.** Cron more frequent than once per day needs a paid plan; deployment protection will block the Slack webhook until you allow it; the connector setup can leave you with a stack of duplicate Slack apps. I kept shipping through these, and each can take an afternoon you did not plan for.

## The one question that decides it

eve is opinionated and Vercel-native by design. That can be a strength or a tax. Your answer depends on **where your agent already runs.**

If your stack runs on Vercel and the AI SDK, eve's conventions will feel like home, the AI Gateway lets you route model calls without putting a provider key in each project, and you'll have a deployed agent the same day. The opinions work for you. This is the clear "yes" lane, and it's the lane I'm in. It's also the lane where the separation-of-concerns payoff shows up: because each bot is its own directory on a shared substrate, the fleet keeps growing without the add-a-feature-break-a-feature tax that caught up with my hand-built system.

For teams that need multi-cloud, self-hosting, or provider portability, eve's Vercel-native assumptions cost more than they help. Durable execution runs on Vercel Workflows, the sandbox on Vercel Sandbox, connectors on Vercel Connect. That tight integration makes it good and makes it a poor fit if leaving Vercel is a requirement. A more portable framework or a lower-level library you wire yourself will serve you better here, at the cost of writing the plumbing eve gives you for free. The name that comes up most in the same breath is [Mastra](https://mastra.ai): also TypeScript, also agents/tools/workflows, with portability across clouds. You gain the freedom to self-host or run multi-cloud; you give up eve's advantage, the managed Vercel Connect / Sandbox / Workflows substrate you'd now assemble and operate yourself. If "we cannot be locked to Vercel" is a hard requirement, that's your fork in the road. If it isn't, eve's tighter integration is the reason to pick it.

**eve trades portability for production-readiness on Vercel.** The quality of that trade depends on whether you planned to be on Vercel.

## Quick verdict by who you are

- **Solo builder / small team already on Vercel:** An emphatic yes, and the strongest case. You get production agents in a day, you spend your time on the bot's job instead of on plumbing, and the directory-per-bot structure keeps the system from regressing on itself as you add capabilities. This is me, and it's the lesson from my last build made into a default. Start with the build post below and copy the directory shape before you customize anything.
- **Enterprise on Vercel wanting governance and observability:** Strong yes, with eyes open on the beta status. The enterprise controls and Agent Runs are there, but pin versions and pilot before you bet a critical workflow on it.
- **Team that needs multi-cloud or self-hosting:** Poor fit by design today, and likely a poor fit long term. Look at a portable framework (see the Mastra comparison above). eve targets Vercel-native agents, which is fine.
- **Someone who wants to learn agent internals from the ground up:** Skip it for that goal. eve's value is hiding the substrate. If your goal is to *understand* durable execution and connectors, build the hard way once first, then come to eve to stop doing it by hand. (I write up the hard-way lessons as I hit them; the [newsletter](/newsletter) is where those land.)
- **Anyone who needs it rock-stable in production this quarter:** Wait for the churn to settle or pin versions with care. It's a public beta and the API will move.

## Would I run my business on it?

I already do, and after everything in this build I'd make the same call again without hesitation. The fleet is running while I write this: bots handing work to each other over authenticated POSTs, parking for my approval in Slack, surviving deploys without dropping a task [Zack: real detail: how it has held up since the original review: uptime, any incidents, whether beta churn bit again, whether you've added or removed bots]. My running fleet carries more weight than a feature list. The conventions are good enough that I stopped thinking about the framework and started thinking about the bots, which is what I want from infrastructure: boring enough that I stop naming it in my commits.

My answer stays yes. I built the capable-but-tangled version once, felt the separation-of-concerns tax first-hand, and eve makes the clean version the default. If you're on Vercel and you want real agents in production without hand-building and then policing that substrate yourself, I'd reach for it again tomorrow. If your stack runs elsewhere, eve remains a good framework and asks you to adopt a platform; that trade belongs to your infrastructure plan.
